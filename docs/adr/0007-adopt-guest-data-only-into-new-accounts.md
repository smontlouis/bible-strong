# Adopt guest data only into newly created accounts

Guest data may be assigned only to the genuinely new account established by the credential operation that created it; existing accounts, provider links, and restored sessions never receive guest data automatically. Account entry must classify ownership before live hydration begins, preserve the existing account-sync boundary through a versioned allowlist, keep unknown classifications non-destructive, and bind any resumable adoption snapshot permanently to its destination UID. This preserves first-account work without weakening cross-account isolation.

Adoption is intentionally lightweight: it reuses stable entity identifiers and existing sync contracts, rebuilds derived system relations and indexes, records a single pending/completed checkpoint, and replays the same snapshot after interruption. The pre-entry backup is best effort and follows normal backup rotation; its failure does not block account entry. A pending snapshot remains visible, is permanently bound to its destination UID, and may resume only for that account. No runtime feature flag or dedicated progress UI is required, but a durable pending failure receives a non-blocking message.

Diagnostics record classification, lifecycle state, provider, duration, failure stage, and aggregate counts without user content, URLs, Bible references, or email addresses. Release requires automated coverage of classification, allowlisting, new-versus-existing ownership, replay, UID isolation, and interruption recovery, followed by native provider smoke tests against the development Firebase environment.
